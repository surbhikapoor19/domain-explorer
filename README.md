# Domain Explorer

Interactive, AI-in-the-loop web app for exploring and comparing robotics research methods, built for the [COMPARE ecosystem](https://robot-manipulation.org) at WPI (NSF POSE funded).

## What it is

Domain Explorer is a dashboard for browsing a corpus of robotics research and asking it questions in plain language. The default domain, **grasp planning**, covers 56 methods drawn from a 55-paper corpus, plus 2,114 benchmark results extracted directly from paper tables. It combines a natural-language copilot (retrieval-augmented generation grounded on a knowledge graph) with a queryable benchmark table and a UMAP similarity map of methods.

The app is **domain-agnostic**: every page is driven by a per-domain `domain-config.json` plus precomputed static JSON, so a second domain (motion planning, "Motion Explorer") runs from the exact same codebase. The React frontend and its serverless API deploy on Vercel; a Python pipeline precomputes all data and is triggered through GitHub Actions. The app is designed to be embedded via iframe (e.g. on robot-manipulation.org).

## Features / Pages

### Graph Reasoning (default landing page)
The AI copilot and knowledge-graph surface. Ask a question and get an answer generated by client-side RAG (hybrid neural MiniLM + BM25 retrieval over a chunked corpus), grounded on the knowledge graph and extracted benchmark facts, with citations to source papers (inline PDF viewer, citation modal, proof blocks). Also renders an interactive knowledge graph with typed edges (`cites`, `outperforms`, `trained_on`, `uses_hardware`, ...) and the method table. This is where every question lands.

### Benchmarks
A flat, tag-filterable view of 2,114 extracted benchmark results across 46 papers. Every number shows its full protocol tags, an evidence grade (A = corroborated, B = single-source, C = low-confidence), and its provenance — each row links to a cropped image of the exact source table. Also holds corroboration-joined leaderboards and pairwise comparisons. No ranking spin: the mostly-B evidence distribution is shown honestly.

### Explorer (toggleable, off by default)
A UMAP scatter of every method positioned by similarity across weighted attributes, with color-by options (cluster, method, training data, hardware, etc.), interactive weight sliders that recompute the UMAP live in the browser, and alternate "Method Connections" (network) and "Cluster Relations" views. Hidden by default; a maintainer enables it site-wide via `explorerEnabled` in the domain config (which ships with the data, so it works in embeds too), with a `?explorer=1` URL override and a per-browser `explorer-enabled` toggle on the Admin page.

### Admin (token-gated, `/admin`)
Maintainer console, not in the public nav. Upload a domain CSV + PDFs (pushed to GitHub LFS), get an LLM-proposed column-to-role mapping, trigger GitHub Actions domain builds, poll build status, switch the active domain, and toggle the Explorer page. Every admin API call requires the `ADMIN_TOKEN` header.

## Architecture

- **Frontend** (`dashboard/src`) — React 19 single-page app built with Create React App via craco. `App.js` handles path-based domain detection and page routing; `DomainContext.js` and `branding.js` carry domain config and branding through the tree. Visualizations use plotly.js, cytoscape, react-force-graph, umap-js, and ml-kmeans.
- **Client AI / RAG** (`dashboard/src/lib`) — the copilot's retrieval and prompt assembly run in the browser: hybrid neural (`@xenova/transformers` MiniLM) + BM25 retrieval, grounded on the knowledge graph and benchmark comparisons. Only the final LLM request leaves the client.
- **Serverless API** (`dashboard/api`, Vercel functions) — `chat.js` is a hardened LLM proxy that keeps provider keys server-side, with per-IP rate limiting and a same-origin allowlist; **Gemini** (`gemini-flash-latest`) is the default provider with automatic **Groq** (`openai/gpt-oss-120b`) fallback on quota errors, 5xx, or empty completions. `spell.js` is a spell-correction proxy. `api/admin/` holds the token-gated maintainer endpoints (upload, trigger-build, build-status, switch-domain, propose-yaml).
- **Data pipeline** (`dashboard/scripts/precompute`, Python) — a package run as `python -m scripts.precompute` that builds the UMAP/explorer data, the RAG chunk corpus, and the knowledge graph. A separate benchmark sub-pipeline (`scripts/precompute/benchmarks`) extracts, normalizes, and aggregates results from paper tables into evidence-graded benchmark data.
- **Static data** (`dashboard/public/data-<domain>/`) — the precomputed JSON that the frontend reads directly: UMAP coordinates, TF-IDF matrices, embeddings, the knowledge graph, RAG chunks, benchmark comparisons, and cropped table images.
- **Build automation** (`.github/workflows`) — `domain-build.yml` runs the full precompute pipeline on `repository_dispatch` and commits the regenerated data; `sheet-poll.yml` nightly-polls each domain's source CSV and triggers a precompute-only rebuild when it changes.

## Getting started / local dev

All commands run from `dashboard/` (the deployed app root).

```bash
cd dashboard
npm install                 # install frontend dependencies

npm start                   # craco start — dev server on http://localhost:3000
                             # (frontend only; /api serverless functions are not available)

source .env.local && vercel dev --listen 3002
                             # run the SPA together with the /api functions locally
                             # (env vars must be sourced inline for the functions to see them)

npm run build                # CI=false craco build — production bundle into dashboard/build
npm test                     # craco test (Jest + React Testing Library)
```

Required env vars in `dashboard/.env.local`:

```
GEMINI_API_KEY=...   # copilot LLM (default provider)
GROQ_API_KEY=...     # copilot LLM (fallback provider)
ADMIN_TOKEN=...      # required by /api/admin/* endpoints
GH_PAT=...           # trigger GitHub Actions builds / LFS upload
```

To regenerate the precomputed data locally (Python, run from `dashboard/`):

```bash
python -m scripts.precompute                                          # build all pages, default (grasp) domain
python -m scripts.precompute --page explorer graph                    # build selected pages only
python -m scripts.precompute --domain domains/motion_planning.yaml    # build a different domain
```

## Deployment

Deployed on Vercel with the project root set to `dashboard/`. The React app builds via craco, and everything in `dashboard/api/` deploys as Vercel serverless functions. `dashboard/vercel.json` rewrites drive path-based SPA routing (`/`, `/admin`, `/:domain`) so client-side routes survive a hard reload. `.vercelignore` excludes `datasets/`, `backend/`, `hgt/`, `docs/`, `scripts/`, and `.github/` from CLI deploys to stay under Vercel's size limits.

Runtime secrets are set as Vercel environment variables: `GEMINI_API_KEY`, `GROQ_API_KEY` (plus optional `GEMINI_MODEL`/`GROQ_MODEL` overrides), `ADMIN_TOKEN`, `GH_PAT`, `ALLOWED_ORIGINS`.

Data is **not** rebuilt on deploy — it's regenerated by the GitHub Actions pipeline (triggered via `repository_dispatch`), which commits the new JSON to the repo; Vercel then redeploys from that commit.

## Data pipeline

`dashboard/scripts/precompute` is the Python pipeline that produces every static JSON the frontend reads. The core pages — `explorer`, `rag`, `graph` — build the UMAP projection, TF-IDF/embeddings, the chunked RAG corpus, and the knowledge graph from GROBID/TEI-parsed papers. The benchmark pipeline (`scripts/precompute/benchmarks`) is a separate multi-stage flow: it locates result tables and extracts cells (Docling TableFormer, TEI table parsing, and an Anthropic Sonnet-vision extractor), renders a cropped image of each source table, normalizes metrics/protocols/units, and joins corroborating reports into evidence-graded (A/B/C) benchmark results and leaderboards.

In production this all runs inside GitHub Actions (`domain-build.yml`) on `repository_dispatch`, which installs GROBID and the required Python dependencies, runs the build, and commits the regenerated data.

## Domains

- **`grasp_planning`** — "Grasp Explorer" (default): 56 methods, 55-paper corpus, full knowledge graph, 2,114 benchmark results across 46 papers. Served at `/grasp-planning`.
- **`motion_planning`** — "Motion Explorer": motion-planning algorithms with their own `domain-config.json` and precomputed data. Served at `/motion-planning`.

Domains are fully config-driven — each has a `public/data-<domain>/domain-config.json` (column roles, default weights, table/color-by columns, priority dimensions, branding) and a matching YAML consumed by the Python pipeline. New domains are onboarded through the Admin page: upload a CSV + PDFs, review the LLM-proposed column-to-role mapping, and trigger a GitHub Actions build — no code changes required.

## Embedding

The app is designed to be embedded on other sites (e.g. robot-manipulation.org) via an iframe. It detects when it's running inside an iframe and hides internal-only controls — the Admin link and the AI Settings gear — from the embedded surface. Embedded sessions cache the last query and its result in `sessionStorage`, keyed by domain, so navigation or reloads inside the iframe restore state, and the viz uses cooperative scroll/zoom (Cmd/Ctrl+scroll) so it doesn't hijack the host page's scrolling. Because `localStorage` does not cross the iframe boundary, the Explorer tab's visibility in embeds is driven by the `explorerEnabled` flag in the domain config (which ships with the data) or a `?explorer=1` parameter on the iframe URL — the Admin toggle is a per-browser override for the standalone app.
