Domain Explorer

Point it at a folder of research papers and a spreadsheet of items, and it turns them into an interactive knowledge map — a searchable graph, evidence-graded benchmark tables, and an AI copilot that answers questions with sourced, verifiable numbers.

How to read this guide In a hurry? The Quick start below gets you to answers in about two minutes.
New here, or coming from a robotics (not ML) background? Read sections 1–7 (“Use it”) top to bottom — that's a complete, jargon-light picture of what the tool does and how to use it. You can safely skip every “↗” deep-dive link and every collapsed panel; nothing essential is hidden inside them.
Implementing, extending, or evaluating it? Sections 8–10 (“Build it”) plus the expandable internals and the Further reading links go all the way down to the papers and source of every component.

1 Get answers in 2 minutes

Six things a robotics engineer actually comes here to do, as recipes. Everything else in this manual is depth behind these.

Four terms, one line each RAG (retrieval-augmented generation) — before answering, the copilot pulls real passages out of the papers and may only write from those, so answers cite sources instead of inventing.
KG (knowledge graph) — a network of typed facts extracted from the corpus: methods, papers, techniques, hardware, and the stated links between them.
TEI — the structured XML each paper's PDF is parsed into (sections, references, citations) so software can read it like a document, not a picture.
Evidence grades A / B / C — how well-corroborated an extracted number is: A2+ independent papers report consistent values Ba single solid source Clow-confidence extraction, or papers disagree.

Find methods for your setup

  1. On Graph Reasoning (the landing page), type your constraint into the ask bar — e.g. which methods use suction grippers in cluttered scenes? — and hit Ask.
  2. The answer names the matching methods with [n] citations; attribute matches (hardware, scene, input type…) are checked against the whole spreadsheet, not just the retrieved papers.
  3. Cross-check in the method table — hover any row to spotlight that method in the graph and every panel.

Compare two methods

  1. Ask compare X vs Y. Methods you name are always pulled in with their own evidence — even if retrieval alone would have missed one.
  2. Read the answer, then the priority-dimension comparison table under it (shared / differs / partial / gap pill per row).
  3. Switch to Benchmarks — the page syncs itself to the methods named in your query (a banner says so; an amber notice appears instead if none of them has an extracted result).

Check where a number came from

  1. Click any [n] citation chip in a copilot answer (works on both Graph Reasoning and Benchmarks).
  2. A source modal opens: the claim being cited (“Cited for …”), the retrieved passage that best matches it with the shared terms highlighted, and its section + page.
  3. Check the badge: supporting passage = the source states the claim; weak match = an explicit warning that the source doesn't clearly state it — treat that citation with caution. [B#] chips open the verified benchmark block instead of a paper passage.

Find benchmark results that match your protocol

  1. Open the Benchmarks tab. In the left filter rail, pick your metric, scene, camera view, success criterion, dataset… Tags in different categories must all hold (AND); tags within one category match either (OR). The count beside each option = how many results picking it would give you.
  2. On any card, click Source — per-value provenance plus a crop of the paper's own table.
  3. Click the crop to enlarge it full-screen (Esc closes).

See everything about one method

  1. On Benchmarks, type its name in the search box (or pick it in the Method facet), then toggle Group by method so all of its evidence reads as one block.
  2. Back on Graph Reasoning, hover its row in the method table — the same method lights up in the graph and every panel; click its graph node for a detail panel.

Spot an open research gap

  1. On the Graph Reasoning landing, click the “interactive panels below” jump cue (or scroll) to Research Gaps.
  2. Pick any two spreadsheet attributes as rows/columns — empty cells are combinations no method in the corpus addresses yet.

2 Overview

Domain Explorer is domain-agnostic. The screenshots people first see are about robotic grasp planning, but nothing in the tool is grasp-specific — it ships with example domains (grasp planning, motion planning) and you can add your own. Wherever this manual says “items”, read it as the things your corpus is about: methods, algorithms, drugs, materials, models…

1 · Ask & reason

Type a natural-language question. The system retrieves relevant passages, walks a knowledge graph, and an LLM synthesizes a grounded answer with a step-by-step proof of how it got there.

2 · Compare every item

A sortable, filterable table lists every item in the corpus with its key attributes side by side, cross-linked to the graph — hover a row and the same item lights up everywhere.

3 · Trust the numbers

Benchmark tables are extracted from the papers' own result tables, each number carrying an evidence grade (A/B/C), a consistency score, and a crop of the source table you can inspect.

4 · Discover the non-obvious

A graph neural network overlays predicted relationships with confidence scores, surfacing connections the papers never stated explicitly.

The one-sentence model

PDFs + a CSV of items  →  extraction & embeddings  →  a knowledge graph and graded benchmarks  →  an explorable dashboard + a copilot. Swap the PDFs and CSV, rebuild, and the whole experience re-skins to your field.

3 Getting started

No install needed to use a published domain — it's a website. Here's the 60-second orientation.

  1. Open the dashboard. You land on Graph Reasoning (the default view).
  2. Either ask a question in the box at the top — until your first answer, a purpose line and clickable example questions sit under it to get you going — or just explore the knowledge graph and the insight panels below it.
  3. The nav tabs switch between Graph Reasoning and Benchmarks. The header's right side holds the Evidence filter, ⚙ Settings, Admin (token-gated, maintainers only), and this Manual. ⚙ and Admin are hidden when the dashboard is embedded on another site; Admin stays reachable via the link in this manual's footer.
  4. The Evidence filter hides low-grade extracted numbers app-wide — precisely: All shows everything, Hide weak hides grade C, Strong only hides both B and C (grade A only).
  5. To zoom the graph, hold Cmd (Mac) / Ctrl (Windows) while scrolling — a plain scroll moves the page instead (a hint appears if you forget the modifier).
  6. To run the copilot on your own key, open ⚙ Settings and pick a provider (see The AI copilot).
Tip Hover any method/item row in the table or any node in the graph — it spotlights the same item across the table, the graph, and every panel. That cross-highlighting is the fastest way to learn how things connect.

5 Key concepts

A few ideas explained once, used everywhere. Expand any to go deeper.

Evidence grades & confidence (A / B / C)
Extracted numbers are not equal. A grade encodes how well-corroborated one is (not how good the method is): A when 2+ independent papers report consistent values for the same method × metric × protocol, B for a single solid source (one paper can only ever be a B — one paper is not corroboration), C for a low-confidence extraction or when papers report the same cell but disagree. A single 0–1 confidence score backs the grades (A≈0.9, B≈0.78, C≈0.45). The header's global Evidence filter uses it app-wide — precisely: All shows everything, Hide weak hides grade C, Strong only hides both B and C. On the Benchmarks page you can also filter by grade directly with the Evidence grade facet in the rail.
Cross-highlighting (“everything connects to everything”)
Hovering an item in any surface — table row, graph node, insight bar — lights up that item everywhere else. Clicking a value in a landing panel (a technique, a timeline year, a most-referenced paper) locks the selection: the graph zooms to the highlighted cohort and a selection bar names the basis and the co-selected methods, with a Clear button. Click empty space, press Esc, or re-click to clear.
Predicted relationships (a graph neural network)
Beyond the relationships literally stated in papers, a heterogeneous graph neural network (GraphSAGE + DistMult — see Pipeline architecture) learns the structure of the corpus and predicts likely-but-undocumented links (e.g. “these two items should probably be compared on the same benchmark”), each with a confidence score and a min-confidence slider. It's a hypothesis generator, clearly separated from documented facts.
Provenance & proof
Every benchmark number carries its lineage: the source paper, the raw cell text, the table caption, the page, and an image crop of the table it came from. Click Source on any benchmark card to verify the number with your own eyes (click the crop to enlarge). In copilot answers, every [n] citation chip opens the passage that best supports that specific claim — and says weak match when it doesn't clearly state it. No black boxes.
Research-gap matrix
A cross-tab of any two attributes from your CSV. Empty cells are coverage gaps — combinations no item in the corpus addresses yet. A fast way to spot open problems / under-explored regions in your field.

6 The AI copilot

The ask bar at the top of every page (answers render on Graph Reasoning and Benchmarks). It runs in your browser: retrieval, graph traversal, and benchmark grounding all happen locally (client-side), and only the final assembled prompt is sent to an LLM.

What you'll see while it works

  • First run: a purpose line + clickable example questions under the ask bar, gone after your first answer.
  • Staged progress under the bar while it generates: Retrieving evidence → Checking benchmarks & facts → Writing the answer.
  • Spell correction: typos in method or technique names are fixed locally and the answer notes “Showing results for …” with what you originally typed.

Reading the answer

  • Citations you can check: click any [n] chip — on Graph Reasoning or Benchmarks — and a source modal opens with the claim being cited (“Cited for …”), the best-matching retrieved passage with shared terms highlighted, its section + page, and a supporting passage vs weak match badge (weak = the source doesn't clearly state the claim; the modal warns you). [B#] chips cite verified benchmark blocks instead of paper passages.
  • Numbers are enforced, not trusted: if any figure in the answer can't be matched to the retrieved sources, a visible “⚠ Verification note” flags it at the end of the answer.
  • Methods you name are never dropped: ask “compare GraspQP to GraspVLA” and both get their own evidence and a slot in the comparison table, even if retrieval alone would have favored one.
  • Overview questions survey everything: “give an overview of the methods” / “what approaches exist?” enumerates the whole method set grouped by approach, instead of picking a couple.
  • Follow-ups work: the previous turn is context, so “which of them is fastest?” resolves the pronoun.
  • Deterministic by design: the same question gives the same answer within a session (per-session answer cache + temperature 0) — see the FAQ.
  • Degraded, not dead: if the AI is unavailable, you still get a data-only answer assembled from the corpus facts, attribute matches, and verified benchmark values — clearly labeled as such.

Bring your own key (run it locally / privately)

Open ⚙ Settings → Provider. The Default option uses a server-side key (production). For local or private use, pick a BYOK provider and paste a token — it's stored only in your browser and sent straight to the provider:

7 Troubleshooting & FAQ

The questions that actually come up, with the honest answers.

The graph doesn't zoom when I scroll.
By design. Hold Cmd (Mac) or Ctrl (Windows) while scrolling to zoom; a plain scroll moves the page so the graph never traps your mouse wheel. If you scroll over the graph without the modifier, a hint overlay (“Use ⌘/Ctrl + scroll to zoom”) appears for a moment to remind you.
A citation opened a passage that only weakly matches the claim.
The modal says weak match and warns you explicitly. It means the retrieved passage shares too little with the cited claim to plausibly support it — the copilot may have generalized across sources rather than restating one. That's surfaced, not hidden: a supporting passage badge means the source states the claim; a weak match badge means treat that citation with caution and check the full paper (the Paper Evidence panel and its “View PDF” button are one click away).
A method I care about has no benchmark rows.
Its papers report numbers using metrics or protocols outside the extracted result tables, or report results only in image-form tables the extractor couldn't read — a property of the corpus, not a bug. When a copilot query names methods and none of them has an extracted result, the Benchmarks page shows an explicit amber notice saying exactly that (rather than silently showing everything unfiltered). The method still has full coverage in the graph, the method table, and the copilot's paper evidence.
I asked the same question twice and got the identical answer.
By design. Answers are cached per session and generation runs at temperature 0, so the same (or near-identically phrased) question returns the same answer instead of paraphrasing itself into subtle inconsistencies. A reproducible answer is a feature in a tool meant for verification. The cache clears on reload, and invalidates automatically if the corpus changes.
Why are most evidence grades B?
Because B means a single solid source, and one paper can only ever be a B. An A requires 2+ independent papers reporting consistent values for the same method × metric × protocol — and robotics benchmarks are not standardized, so most combinations are reported by exactly one paper. Papers also frequently re-quote each other's baseline numbers instead of re-running them (re-quoting is not corroboration, so it can't earn an A), and when two papers do re-run the same setup they often disagree (→ C). A mostly-B board is the honest picture of the literature, not a data defect.
The answer starts with “The AI summary is temporarily unavailable”.
The language model couldn't be reached (rate limit, outage). The copilot degrades instead of failing: it serves what was computed deterministically without the LLM — corpus facts, the methods matching your query's attributes, and verified benchmark values. Ask again in a minute, or set your own key under ⚙ Settings.

8 Pipeline architecture

How raw PDFs become the dashboard. Every step is precomputed offline into static JSON, so the live site is fast and serverless.

Step by step (with the method behind each stage)
1 · Extract. GROBID parses born-digital text into structured TEI XML (sections, references, in-text citation context, plus the figures/tables/equations/authors/bibliography the graph uses). Docling (IBM Research) parses each PDF's tables: the layout and cell text serve as deterministic ground truth, and bounding boxes mean each number keeps a crop — OCR off, since papers are born-digital.
2 · RAG. Text is split into chunks and embedded — each chunk is turned into a list of numbers that captures its meaning, so passages about similar things sit near each other — using the compact all-MiniLM-L6-v2 sentence-transformer, and stored in a Chroma vector index. At query time the copilot embeds your question in the browser with Transformers.js for semantic search and blends it with Okapi BM25 lexical scoring.
3 · Knowledge graph. Assembled from parallel extraction tracks, each edge labeled with its provenance: the TEI structural parse (figures, tables, equations, authors, bibliography), regex-extracted facts, LLM-extracted entities (contributions, problems, limitations, comparison claims), and CSV metadata plus a citation resolver. The highest-provenance track is the verified-triple extractor: per text chunk, the LLM proposes (relation, object, verbatim quote) triples against a fixed 8-relation schema (uses_backbone, trained_on, uses_hardware, outperforms_claim…), and any triple whose quote is not literally a substring of that chunk is mechanically rejected — the model cannot assert a relation without a quote that survives the check. Surviving triples become edges carrying the quote + chunk id (extraction='verified_llm'), so the claim behind the edge is checkable against the paper text. Prose “outperforms” claims resolve every method the sentence names (acronym- and punctuation-tolerant; the most specific matched alias wins) and become edges only when the claim actually asserts superiority (“outperforms”, “better than”, “surpasses”…) — a bare “unlike X” stays a comparison, not a win.
4 · Graph link prediction. A heterogeneous GraphSAGE graph neural network (made type-aware with PyG's to_hetero) plus a DistMult relation decoder learns the corpus structure and predicts latent links — trained per domain on the schema exported from that domain's knowledge graph. Bibliometric relations (authorship, citations…) stay in the graph for message passing but are excluded from the training loss, and per-relation validation/test metrics (AUC, MRR, Hits@k) are persisted with every trained model. The UI applies a support gate on top: predicted edges for any relation with fewer than 30 real training edges are hidden, and the view says how many were gated (“N low-evidence predictions hidden”) — a model cannot learn a relation from a handful of examples, so those predictions are noise, not insight. (The build step is named hgt for historical reasons; the architecture is GraphSAGE + DistMult, not a Heterogeneous Graph Transformer.)
5 · Precompute. Writes all the static JSON the pages read (items, knowledge-graph, landing, etc.) to public/data-<domain>/, served statically. It also computes a UMAP 2-D layout + HDBSCAN clusters for the Explorer similarity map — an optional legacy view, off by default, that a maintainer can toggle on only from Admin. Clustering is not used for any inference: not in the knowledge graph, not in the copilot.
6 · Benchmarks. Docling's parsed tables are the deterministic ground truth. When a vision key exists (ANTHROPIC_API_KEY, or GROQ_API_KEY via the Groq-vision fallback), a VLM reads each table crop for the semantics flat cell text loses (multi-level headers, units) — and every VLM row is verified against Docling's extracted cell text: a value not found in the cells is flagged unverified at the lowest confidence, so the vision model can miss rows but cannot invent values. Table headers map to metrics through a per-domain registry (the aliases in benchmarks/config/<domain>.json); headers that don't resolve are quarantined with an audit list of the exact raw strings, and propose_aliases.py clusters them and drafts human-reviewable registry proposals — nothing is ever auto-applied. Aggregation (build_benchmarks.py) then emits: (a) results — every value-bearing extraction, cleaned (citation-marker method names stripped; values-as-headers, bare abbreviations, and condition-as-metric labels dropped), each carrying its method (alias-salvaged where possible, flagged unresolved otherwise), metric label, value plus the paper's own value_str, unit, protocol condition tokens, dataset, self- vs third-party reporting, and a per-record grade that inherits the cross-validation grade when 2+ papers corroborate the same cell; (b) derived comparable views — leaderboards (2+ methods on the same metric + protocol), cross-validations, and graded pairwise comparisons; (c) quarantine stats. The Benchmarks page renders results — a filterable table, explicitly not a leaderboard. The table-derived comparisons also enrich the knowledge graph: one graded outperforms edge per (winner, loser) paper pair, aggregated over every metric/condition the pair was compared on (the edge carries n_comparisons, the metric list, and the pair's best grade; extraction='benchmark_v2'). This step runs after precompute and is triggered separately (the Admin “Build benchmarks” action / the benchmark step).
How a build runs (CI)
Domain builds run in GitHub Actions via repository_dispatch — the Admin panel's Build button sends the event. The payload selects the step chain: grobid,rag,kg,hgt,precompute (main build) or benchmark; each step skips when its output already exists, and client_payload.force re-runs those steps anyway (e.g. after a pipeline code change). PDFs ship in the repo as datasets/<slug>/papers.zip via Git LFS; the workflow pulls and unzips them, runs scripts/ingest_domain.py, and commits the outputs (TEI, extracted facts/entities, verified triples, the knowledge graph, the link-prediction schema, and all dashboard/public/data-<slug>/ JSON) back to main. LLM-dependent steps — entity extraction, verified triples, Groq-vision table reading — require the GROQ_API_KEY Actions secret and degrade gracefully without it: they're skipped (or extraction runs born-digital-only), so a keyless build still produces a working, less-enriched domain.
How the knowledge graph is built — and how edges work across domains
The graph is assembled from four extraction tracks, then normalized:
  • Your CSV (+ a citation resolver) → one item node per row, plus an attribute node for each categorical value (a technique, a piece of hardware, a dataset, a year…).
  • GROBID / TEI (paper structure) → paper, author, institution, year, figure, table, equation and reference nodes.
  • Regex facts + LLM entities over the paper text → contribution, problem, limitation, comparison and method-step nodes.
  • Verified triples (highest provenance) → edges whose claim carries a verbatim, mechanically checked source quote (see step 3 above).
Edges are typed, and the edge vocabulary is shared across every domain — e.g. described_in, uses_hardware, trained_on, implemented_in, maintained_by, addresses_problem, contributes, compares/outperforms, cites_external, authored_by, co_authored_with, published_in_year, has_figure/table/equation, plus computed semantically_similar / similar_in_role links. An outperforms edge additionally names how it was earned: prose_claim (a superiority sentence), verified_llm (quote-verified claim), or benchmark_v2 (table-derived, graded, aggregated per paper pair).

The cross-domain trick: CSV edges are role-driven, not column-name-driven. Each column is mapped to a semantic role (e.g. hardware.platform, train.regime, meta.maintainer, identity.year), and the role — not the literal header — decides the edge type. So a grasp sheet's “End-effector Hardware” column and a motion sheet's “Middleware” column each become the appropriate relationship to their values; a new domain just maps its own columns to roles and the same machinery yields the same edge semantics. Paper-structure edges come from TEI, claim edges from the LLM, and similarity edges from the chunk embeddings.

Finally, per-domain alias tables (Admin → Knowledge-Graph aliases) canonicalize surface forms (rrt*→RRT*, franka→Franka Emika Panda) so one technique / robot / problem isn't fragmented into several nodes.

10 Add your own domain

This is the whole point: the dashboard is a template over a corpus. To bring a new field in, you supply your corpus (a CSV + the PDFs) plus a small one-time vocabulary block, then rebuild.

① A CSV of items

One row per item, columns for the attributes you want to explore and compare (the item table and research-gap matrix read these directly).

② The PDFs

The papers describing those items. Everything downstream — graph, embeddings, benchmarks — is derived from them.

  1. In Admin, create the domain and upload the CSV + PDFs. An LLM proposes a column→role mapping (title, description, attributes) from your headers; you confirm or adjust it — no code.
  2. In the same editor, optionally provide the small amount of domain vocabulary the intelligence layer needs: Knowledge-Graph aliases (normalize spelling variants of your techniques / hardware / problems to one canonical label) and Benchmark metrics (the metrics, conditions, and datasets to read from result tables). The copilot's ranking vocabulary is then auto-derived from those metric aliases — nothing extra to type. Optionally paste the domain's Google Drive folder of CSV exports to enable the nightly auto-sync (see Keep it live below).
  3. Hit Build. CI runs the main chain (extraction → RAG → graph → link prediction → precompute) and commits the precomputed JSON back. Benchmarks are a separate Build benchmarks action (the Docling + VLM-verified table extractor, run after precompute), so trigger that too once the main build succeeds.
  4. When both finish, your domain is a first-class, interactive view — the copilot, benchmarks, and graph re-skin to your field automatically.
Keep it live (nightly sync) Give a domain a Google Drive folder of CSV exports (set it in Admin when you build the domain). A nightly job pulls the newest CSV from that folder, hash-compares it to the current one, and — only when it changed — commits it and triggers a precompute-only rebuild (no PDF re-ingest; unchanged folders are left untouched). That rebuild recomputes everything CSV-derived: the methods table, Research Gaps, Timeline, Benchmarks, and the optional UMAP similarity-map layout. The RAG/copilot corpus and the paper-extracted plots (Most Referenced, Techniques) are preserved unchanged, and the knowledge-graph keeps its existing nodes — a brand-new item gains a graph node only after a full Admin build (which re-ingests its PDF). PDFs are re-ingested only when you trigger that full build.
Safe rebuilds Every precomputed artifact is written through an overwrite guard: a failed or partial rebuild (missing data, a crashed step) can never blank a populated file — the last good data is kept.

What's automatic vs what you provide

Automatic from your CSV + PDFsYou provide once (in Admin)
Branding, column→role rendering, the item table, knowledge-graph structure, RAG retrieval, evidence grades, and the copilot's method summaries. Knowledge-graph alias tables (spelling normalization) and the benchmark metrics + conditions + datasets. The copilot's ranking keywords are derived from these — not typed separately.

How domain-agnostic is it really? (the honest version)

The presentation layer is genuinely agnostic — it reads everything from config, so the chrome re-skins for any field with zero code changes. The intelligence layer (copilot vocabulary, graph entity-normalization, benchmark metrics) is config-driven too, but those configs encode real domain knowledge you supply once in Admin — they are not inferred for free. So the honest claim is “domain-adaptable with a small, one-time vocabulary file,” not “drop in PDFs and everything is perfect.” A second domain — motion planning — is configured end-to-end as the working proof case: its knowledge graph, benchmarks, and copilot all run from its own config via exactly the path your field would follow. The richness of the benchmarks still depends on whether your papers report comparable result tables — that is a property of the corpus, not the engine.

Extraction (PDF → structured data)

  • GROBIDMachine-learning tool that parses a scientific-paper PDF into structured TEI XML — title, sections, references, in-text citations.
  • TEI — Text Encoding InitiativeThe open XML standard GROBID emits, so papers are stored in a consistent, machine-readable structure. (P5 Guidelines)
  • DoclingIBM Research tool that extracts result tables + layout with bounding boxes, so every number keeps an image crop of its source cell. (technical report)

Retrieval & the copilot

  • all-MiniLM-L6-v2The compact sentence-transformer that turns each text chunk into a 384-number vector for meaning-based search. (library: SBERT / sentence-transformers)
  • Transformers.jsRuns that same embedding model in the browser, so your question is embedded client-side at query time.
  • ChromaOpen-source vector database that stores the embeddings for similarity retrieval.
  • Okapi BM25The classic keyword-relevance score blended with semantic search; the definitive monograph by its creator (Robertson & Zaragoza, 2009).
  • Hugging Face · OpenRouter · GroqThe bring-your-own-key LLM providers the copilot can call to synthesize the final answer.

Graph & machine learning

  • GraphSAGEThe graph neural network that learns the corpus structure; applied to the typed knowledge graph (via PyG's to_hetero) to predict latent links. (Hamilton, Ying & Leskovec, 2017.)
  • DistMultThe relation decoder that scores how likely a typed edge (e.g. “outperforms”) is between two nodes. (Yang et al., 2015.)
  • PyTorch Geometric (PyG)The graph-deep-learning framework the link-prediction model is built and trained in, on top of PyTorch.
  • UMAPDimensionality reduction that projects items into an optional 2-D similarity map computed during precompute. (McInnes, Healy & Melville)
  • HDBSCANDensity-based clustering that colours the natural families of items on that map.

App & infrastructure

  • React (Create React App)The single-page dashboard UI.
  • Cytoscape.jsRenders the interactive knowledge graph and the predicted-edge view.
  • Plotly.jsDraws the similarity-map scatter in the optional legacy Explorer view (off by default; toggleable from Admin).
  • VercelHosts the static site plus the serverless API functions (admin, chat proxy).
  • GitHub ActionsRuns the per-domain ingestion pipeline triggered from the Admin page.
