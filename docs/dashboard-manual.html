Domain Explorer

Point it at a folder of research papers and a spreadsheet of items, and it turns them into an interactive knowledge map — a searchable graph, evidence-graded benchmark tables, and an AI copilot that answers questions with sourced, verifiable numbers.

How to read this manual New here, or coming from a robotics (not ML) background? Read sections 1–5 (“Use it”) top to bottom — that's a complete, jargon-light picture of what the tool does and how to use it. You can safely skip every “↗” deep-dive link and every collapsed panel; nothing essential is hidden inside them.
Implementing, extending, or evaluating it? Sections 6–8 (“Build it”) plus the expandable internals and the Further reading links go all the way down to the papers and source of every component.

1 Overview

Domain Explorer is domain-agnostic. The screenshots people first see are about robotic grasp planning, but nothing in the tool is grasp-specific — it ships with example domains (grasp planning, motion planning) and you can add your own. Wherever this manual says “items”, read it as the things your corpus is about: methods, algorithms, drugs, materials, models…

1 · Ask & reason

Type a natural-language question. The system retrieves relevant passages, walks a knowledge graph, and an LLM synthesizes a grounded answer with a step-by-step proof of how it got there.

2 · Compare every item

A sortable, filterable table lists every item in the corpus with its key attributes side by side, cross-linked to the graph — hover a row and the same item lights up everywhere.

3 · Trust the numbers

Benchmark tables are extracted from the papers' own result tables, each number carrying an evidence grade (A/B/C), a consistency score, and a crop of the source table you can inspect.

4 · Discover the non-obvious

A graph neural network overlays predicted relationships with confidence scores, surfacing connections the papers never stated explicitly.

The one-sentence model

PDFs + a CSV of items  →  extraction & embeddings  →  a knowledge graph and graded benchmarks  →  an explorable dashboard + a copilot. Swap the PDFs and CSV, rebuild, and the whole experience re-skins to your field.

2 Getting started

No install needed to use a published domain — it's a website. Here's the 60-second orientation.

  1. Open the dashboard. You land on Graph Reasoning (the default view).
  2. Either ask a question in the box at the top, or just explore the knowledge graph and the insight panels below it.
  3. Use the top nav to switch views: Graph Reasoning, Benchmarks, and Admin (token-gated — it asks for an access token before opening).
  4. Use the Evidence filter in the header (All / Hide weak / Strong only) to hide weak/low-grade numbers across the whole app.
  5. To run the copilot on your own key, open ⚙ Settings and pick a provider (see The AI copilot).
Tip Hover any method/item row in the table or any node in the graph — it spotlights the same item across the table, the graph, and every panel. That cross-highlighting is the fastest way to learn how things connect.

4 Key concepts

A few ideas explained once, used everywhere. Expand any to go deeper.

Evidence grades & confidence (A / B / C)
Extracted numbers are not equal. A grade encodes how much to trust one: A when multiple independent papers report consistent values, B for a single solid source, C when a number is weak or papers disagree. A single 0–1 confidence score (grade + a penalty for high variance) backs the grades. The header's global Evidence filter (All / Hide weak / Strong only) hides weak / grade-C numbers in the copilot's answers and the detail panels; on the Benchmarks page you filter by grade directly with the Evidence grade facet in the rail.
Cross-highlighting (“everything connects to everything”)
Hovering or selecting an item in any surface — table row, graph node, scatter dot, insight bar — lights up that item everywhere else. Clicking a category (a technique, a year, an author) highlights every item connected to it across all panels. Click empty space, press Esc, or re-click to clear.
Predicted relationships (a graph neural network)
Beyond the relationships literally stated in papers, a heterogeneous graph neural network learns the structure of the corpus and predicts likely-but-undocumented links (e.g. “these two items should probably be compared on the same benchmark”), each with a confidence score and a min-confidence slider. It's a hypothesis generator, clearly separated from documented facts.
Provenance & proof
Every benchmark number carries its lineage: the source paper, the raw cell text, the table caption, the page, and an image crop of the table it came from. Click Source on any row to verify the number with your own eyes — no black boxes.
Research-gap matrix
A cross-tab of any two attributes from your CSV. Empty cells are coverage gaps — combinations no item in the corpus addresses yet. A fast way to spot open problems / under-explored regions in your field.

5 The AI copilot

The query box on Graph Reasoning. It runs in your browser: retrieval, graph traversal, and benchmark grounding all happen locally (client-side), and only the final assembled prompt is sent to an LLM.

Bring your own key (run it locally / privately)

Open ⚙ Settings → Provider. The Default option uses a server-side key (production). For local or private use, pick a BYOK provider and paste a token — it's stored only in your browser and sent straight to the provider:

6 Pipeline architecture

How raw PDFs become the dashboard. Every step is precomputed offline into static JSON, so the live site is fast and serverless.

Step by step (with the method behind each stage)
1 · Extract. GROBID parses born-digital text into structured TEI XML (sections, references, in-text citation context). Docling (IBM Research) reads the result tables (with bounding boxes, so each number keeps a crop) — OCR off, since papers are born-digital.
2 · RAG. Text is split into chunks and embedded — each chunk is turned into a list of numbers that captures its meaning, so passages about similar things sit near each other — using the compact all-MiniLM-L6-v2 sentence-transformer, and stored in a Chroma vector index. At query time the copilot embeds your question in the browser with Transformers.js for semantic search and blends it with Okapi BM25 lexical scoring.
3 · Knowledge graph. Entities (items, papers, techniques, hardware, authors, claims) and their relationships — including citation stance — are assembled into a graph.
4 · Graph link prediction. A heterogeneous GraphSAGE graph neural network (made type-aware with PyG's to_hetero) plus a DistMult relation decoder learns the corpus structure and predicts latent links. (The build step is named hgt for historical reasons; the architecture is GraphSAGE + DistMult, not a Heterogeneous Graph Transformer.)
5 · Precompute. Writes all the static JSON the pages read (items, knowledge-graph, landing, etc.) to public/data-<domain>/, served statically. It also computes a UMAP 2-D layout + HDBSCAN clusters for an optional 2-D similarity map (not currently surfaced in the UI). Clustering is not used for any inference: not in the knowledge graph, not in the copilot.
6 · Benchmarks. Table cells are normalized (units, multi-level headers, aliases), de-duplicated per paper, scored for consistency, and assigned A/B/C grades with provenance. This step runs after precompute and is triggered separately (the Admin “Build benchmarks” action / the benchmark step).
How the knowledge graph is built — and how edges work across domains
The graph is assembled from three sources, then normalized:
  • Your CSV → one item node per row, plus an attribute node for each categorical value (a technique, a piece of hardware, a dataset, a year…).
  • GROBID / TEI (paper structure) → paper, author, institution, year, figure, table, equation and reference nodes.
  • LLM extraction over the paper text → contribution, problem, limitation, comparison and method-step nodes.
Edges are typed, and the edge vocabulary is shared across every domain — e.g. described_in, uses_hardware, trained_on, implemented_in, maintained_by, addresses_problem, contributes, compares/outperforms, cites_external, authored_by, co_authored_with, published_in_year, has_figure/table/equation, plus computed semantically_similar / similar_in_role links.

The cross-domain trick: CSV edges are role-driven, not column-name-driven. Each column is mapped to a semantic role (e.g. hardware.platform, train.regime, meta.maintainer, identity.year), and the role — not the literal header — decides the edge type. So a grasp sheet's “End-effector Hardware” column and a motion sheet's “Middleware” column each become the appropriate relationship to their values; a new domain just maps its own columns to roles and the same machinery yields the same edge semantics. Paper-structure edges come from TEI, claim edges from the LLM, and similarity edges from the chunk embeddings.

Finally, per-domain alias tables (Admin → Knowledge-Graph aliases) canonicalize surface forms (rrt*→RRT*, franka→Franka Emika Panda) so one technique / robot / problem isn't fragmented into several nodes.

8 Add your own domain

This is the whole point: the dashboard is a template over a corpus. To bring a new field in, you supply your corpus (a CSV + the PDFs) plus a small one-time vocabulary block, then rebuild.

① A CSV of items

One row per item, columns for the attributes you want to explore and compare (the item table and research-gap matrix read these directly).

② The PDFs

The papers describing those items. Everything downstream — graph, embeddings, benchmarks — is derived from them.

  1. In Admin, create the domain and upload the CSV + PDFs. An LLM proposes a column→role mapping (title, description, attributes) from your headers; you confirm or adjust it — no code.
  2. In the same editor, optionally provide the small amount of domain vocabulary the intelligence layer needs: Knowledge-Graph aliases (normalize spelling variants of your techniques / hardware / problems to one canonical label) and Benchmark metrics (the metrics, conditions, and datasets to read from result tables). The copilot's ranking vocabulary is then auto-derived from those metric aliases — nothing extra to type. Optionally paste the domain's Google Drive folder of CSV exports to enable the nightly auto-sync (see Keep it live below).
  3. Hit Build. CI runs the main chain (extraction → RAG → graph → HGT → precompute) and commits the precomputed JSON back. Benchmarks are a separate Build benchmarks action (the Docling table extractor, run after precompute), so trigger that too once the main build succeeds.
  4. When both finish, your domain is a first-class, interactive view — the copilot, benchmarks, graph, and map re-skin to your field automatically.
Keep it live (nightly sync) Give a domain a Google Drive folder of CSV exports (set it in Admin when you build the domain). A nightly job pulls the newest CSV from that folder, hash-compares it to the current one, and — only when it changed — commits it and triggers a precompute-only rebuild (no PDF re-ingest; unchanged folders are left untouched). That rebuild recomputes everything CSV-derived: the methods table, Research Gaps, Timeline, Benchmarks, and the optional UMAP similarity-map layout. The RAG/copilot corpus and the paper-extracted plots (Most Referenced, Techniques) are preserved unchanged, and the knowledge-graph keeps its existing nodes — a brand-new item gains a graph node only after a full Admin build (which re-ingests its PDF). PDFs are re-ingested only when you trigger that full build.
Safe rebuilds Every precomputed artifact is written through an overwrite guard: a failed or partial rebuild (missing data, a crashed step) can never blank a populated file — the last good data is kept.

What's automatic vs what you provide

Automatic from your CSV + PDFsYou provide once (in Admin)
Branding, column→role rendering, the item table, knowledge-graph structure, RAG retrieval, evidence grades, and the copilot's method summaries. Knowledge-graph alias tables (spelling normalization) and the benchmark metrics + conditions + datasets. The copilot's ranking keywords are derived from these — not typed separately.

How domain-agnostic is it really? (the honest version)

The presentation layer is genuinely agnostic — it reads everything from config, so the chrome re-skins for any field with zero code changes. The intelligence layer (copilot vocabulary, graph entity-normalization, benchmark metrics) is config-driven too, but those configs encode real domain knowledge you supply once in Admin — they are not inferred for free. So the honest claim is “domain-adaptable with a small, one-time vocabulary file,” not “drop in PDFs and everything is perfect.” A second domain — motion planning — is configured end-to-end as the working proof case: its knowledge graph, benchmarks, and copilot all run from its own config via exactly the path your field would follow. The richness of the benchmarks still depends on whether your papers report comparable result tables — that is a property of the corpus, not the engine.

9 Further reading

Every external technology and method this manual names, linked to its authoritative source — the original papers, official docs, and repositories — for a deeper dive and full transparency. This section is optional: the manual stands on its own without clicking any of it.

Extraction (PDF → structured data)

  • GROBIDMachine-learning tool that parses a scientific-paper PDF into structured TEI XML — title, sections, references, in-text citations.
  • TEI — Text Encoding InitiativeThe open XML standard GROBID emits, so papers are stored in a consistent, machine-readable structure. (P5 Guidelines)
  • DoclingIBM Research tool that extracts result tables + layout with bounding boxes, so every number keeps an image crop of its source cell. (technical report)

Retrieval & the copilot

  • all-MiniLM-L6-v2The compact sentence-transformer that turns each text chunk into a 384-number vector for meaning-based search. (library: SBERT / sentence-transformers)
  • Transformers.jsRuns that same embedding model in the browser, so your question is embedded client-side at query time.
  • ChromaOpen-source vector database that stores the embeddings for similarity retrieval.
  • Okapi BM25The classic keyword-relevance score blended with semantic search; the definitive monograph by its creator (Robertson & Zaragoza, 2009).
  • Hugging Face · OpenRouter · GroqThe bring-your-own-key LLM providers the copilot can call to synthesize the final answer.

Graph & machine learning

  • GraphSAGEThe graph neural network that learns the corpus structure; applied to the typed knowledge graph (via PyG's to_hetero) to predict latent links. (Hamilton, Ying & Leskovec, 2017.)
  • DistMultThe relation decoder that scores how likely a typed edge (e.g. “outperforms”) is between two nodes. (Yang et al., 2015.)
  • PyTorch Geometric (PyG)The graph-deep-learning framework the link-prediction model is built and trained in, on top of PyTorch.
  • UMAPDimensionality reduction that projects items into an optional 2-D similarity map computed during precompute. (McInnes, Healy & Melville)
  • HDBSCANDensity-based clustering that colours the natural families of items on that map.

App & infrastructure

  • React (Create React App)The single-page dashboard UI.
  • Cytoscape.jsRenders the interactive knowledge graph and the predicted-edge view.
  • Plotly.jsDraws the scatter / UMAP map and the benchmark charts.
  • VercelHosts the static site plus the serverless API functions (admin, chat proxy).
  • GitHub ActionsRuns the per-domain ingestion pipeline triggered from the Admin page.
